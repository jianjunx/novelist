# 多Agent协作AI小说创作平台 - 设计文档

## 背景

个人创作者（独立作家、网文作者）需要超越简单文本生成的AI辅助工具。他们希望多个AI"角色"——编辑、读者、评论家——像一个创意工作坊一样协作讨论、改进作品。本平台为此工作流提供结构化环境。

**目标**：本地优先开发，Go后端 + React前端，Eino框架支持多模型LLM，PostgreSQL + pgvector存储。

## 整体架构

**数据流向**：
1. 用户请求 → API层 → 编排器
2. 编排器 → 记忆系统组装上下文
3. 记忆系统 → pgvector语义检索相关记忆 → 组装完整上下文
4. 编排器 + 上下文 → Agent层调用
5. Agent层 → 模型适配层 → LLM API
6. 响应 → WebSocket实时推送到前端

## Agent角色定义

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **构思Agent** | 世界观构建、人物设计、故事大纲 | 用户创意描述 | 结构化设定文档 |
| **写作Agent** | 根据大纲和设定生成章节内容 | 大纲+设定+前文 | 章节文本 |
| **编辑Agent** | 审查逻辑一致性、文笔质量、节奏、自然度 | 章节文本+设定 | 优先级排序的建议列表 |
| **读者Agent** | 模拟读者视角，反馈阅读体验 | 章节文本 | 体验反馈 |
| **评论家Agent** | 文学性分析、风格建议 | 章节文本+风格要求 | 深度分析报告 |

### 去AI味策略

不单独设立Agent，而是：
- **编辑Agent**：将"自然度"作为审查维度，检测AI痕迹（过于工整的句式、重复的过渡词、模板化描写）
- **写作Agent**：系统提示中注入去AI味的写作准则，从源头减少问题

## Agent记忆系统

三层记忆架构，确保Agent输出连贯、上下文感知：

### 长期记忆（始终可用）
- 世界观设定（魔法体系、地理、历史）
- 主线剧情大纲
- 核心人物档案（主角、反派、关键配角）
- 风格指南 / 写作规则

### 短期记忆（上下文相关，按章节/场景）
- 当前正在发展的子线剧情
- 当前场景的临时人物
- 近期剧情事件（最近3-5章）
- 当前场景的地点/环境细节

### 工作记忆（当前任务专用）
- 当前讨论中的建议
- 用户最近的修改
- 本次Agent调用的特定上下文

### 上下文组装流程
```
每次Agent调用前，编排器自动组装上下文：
  1. 加载长期记忆（全量，但摘要化）
  2. 语义检索短期记忆（pgvector，基于当前章节内容检索相关设定）
  3. 加载工作记忆（当前任务相关）
  4. 合并 → 注入Agent的系统提示
```

确保每个Agent拥有正确的上下文，同时不超出token限制。

### 运行时记忆管理

**长期记忆**：存储在PostgreSQL，Agent调用时加载。
- characters、world_settings、outlines、chapters表作为长期记忆
- 注入前进行摘要（以保持在token限制内）

**短期记忆**：每个项目会话的内存缓存。
- 最近章节（3-5章）保持在内存中
- 当前章节的相关实体（人物、设定）通过pgvector相似度检索
- 用户编辑设定/人物时缓存失效

**工作记忆**：临时的，每次Agent调用专用。
- 当前讨论建议
- 用户最近的修改（来自WebSocket事件）
- Agent调用完成后丢弃

## 错误处理

| 场景 | 策略 |
|------|------|
| Agent调用失败（API错误） | 重试2次（指数退避），然后向用户报错 |
| 速率限制超出 | 排队请求，向用户显示"等待中"状态 |
| Token限制超出 | 编排器截断上下文（优先级：工作记忆 > 短期记忆 > 长期记忆） |
| Agent返回格式错误 | 解析错误，要求Agent重试并附加格式提醒 |
| WebSocket断开 | 自动重连，从最后检查点恢复流式传输 |

## 讨论工作流

示例：章节审稿流程

```
用户提交章节草稿
  ↓
写作Agent生成内容
  ↓
┌─ 讨论轮次 ─────────────────────────────────────┐
│  编辑Agent: "第3段逻辑问题..."                   │
│  读者Agent: "开头抓人，中间拖沓"                  │
│  评论家Agent: "对话风格不一致"                    │
│  → 按类别汇总建议                                │
│  → 去重相似建议                                  │
│  → 按优先级排序                                  │
└─────────────────────────────────────────────────┘
  ↓
写作Agent根据建议修改
  ↓
用户确认 / 手动编辑
```

**轮次控制**：默认1轮。用户可选择"深度审稿"→ 2-3轮。结果通过WebSocket实时推送到前端。

## 核心工作流

### 1. 构思工作流（多轮对话）
```
用户提供初始创意
  ↓
构思Agent提出澄清问题（类型、风格、语调、核心冲突...）
  ↓
用户回答 → 构思Agent理解细化
  ↓ （重复2-4轮）
构思Agent生成大纲 + 人物档案 + 世界观设定
  ↓
用户通过右侧实时预览区审查
  ↓
用户要求调整 → 构思Agent修改
  ↓ （按需重复）
用户确认 → 存入数据库 → 跳转到项目仪表板
```

### 2. 写作工作流
```
选择章节大纲
  → 语义检索相关设定/人物/前文 (pgvector)
  → 写作Agent生成初稿
  → 进入讨论工作流
  → 写作Agent根据反馈修改
  → 用户确认 → 存入数据库
```

### 3. 讨论工作流
```
输入：章节草稿 + 相关设定上下文
  │
  ├─ 并行调用：编辑Agent、读者Agent、评论家Agent
  │  （每个Agent独立审查，返回结构化建议）
  │
  ├─ 汇总结果
  │  - 按建议类型分组（逻辑、文笔、节奏、风格、自然度）
  │  - 去重相似建议
  │  - 按优先级排序
  │
  ├─ 生成修改方案
  │  - 综合所有建议，生成具体修改指令
  │
  └─ 返回修改方案给写作Agent执行
```

## 数据模型

PostgreSQL + pgvector：

```sql
-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 小说项目
CREATE TABLE projects (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    title TEXT NOT NULL,
    genre TEXT,
    description TEXT,
    style_guide TEXT,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

-- 人物档案
CREATE TABLE characters (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    role TEXT,
    personality TEXT,
    background TEXT,
    appearance TEXT,
    relationships JSONB,
    embedding vector(1536),
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 世界观设定
CREATE TABLE world_settings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    category TEXT NOT NULL,
    content TEXT NOT NULL,
    embedding vector(1536),
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 故事大纲
CREATE TABLE outlines (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    act INT,
    chapter_num INT,
    summary TEXT,
    key_events JSONB,
    status TEXT DEFAULT 'draft',
    embedding vector(1536),
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 章节
CREATE TABLE chapters (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    outline_id UUID REFERENCES outlines(id),
    chapter_num INT NOT NULL,
    title TEXT,
    content TEXT,
    word_count INT DEFAULT 0,
    status TEXT DEFAULT 'draft',
    embedding vector(1536),
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

-- 讨论记录
CREATE TABLE discussions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    chapter_id UUID REFERENCES chapters(id) ON DELETE CASCADE,
    round_num INT NOT NULL,
    agent_role TEXT NOT NULL,
    content TEXT NOT NULL,
    suggestion_type TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 对话记录（构思Agent多轮对话）
CREATE TABLE conversations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    role TEXT NOT NULL,  -- 'user' 或 'agent'
    content TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

**pgvector用途**：
- 写作时语义检索相关设定/人物
- 检索前文内容保持一致性
- 讨论建议去重

## 技术栈

### 后端 (Go)
- **Web框架**：Gin（成熟生态，良好的中间件支持）
- **AI框架**：Eino（Agent编排、模型调用）
- **数据库**：PostgreSQL + pgvector
- **ORM**：GORM（功能丰富，适合快速开发）
- **WebSocket**：gorilla/websocket

### 模型适配层设计

```
┌─────────────────────────────────────────────────────┐
│              模型适配层 (Model Adapter)               │
│                                                     │
│  ┌─────────────────────────────────────────────┐    │
│  │  DeepSeek适配器 (主模型)                    │    │
│  │  ├─ 深度集成DeepSeek API                    │    │
│  │  ├─ 支持DeepSeek特有功能:                   │    │
│  │  │  · 上下文缓存 (Context Caching)          │    │
│  │  │  · 长文本优化 (128K上下文)               │    │
│  │  │  · 中文写作特化提示词                    │    │
│  │  ├─ 流式输出优化                            │    │
│  │  └─ 错误重试与降级策略                      │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  ┌─────────────────────────────────────────────┐    │
│  │  其他模型适配器 (备选)                      │    │
│  │  ├─ Claude适配器                            │    │
│  │  ├─ OpenAI适配器                            │    │
│  │  └─ 本地模型适配器 (Ollama等)               │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

**DeepSeek深度适配要点**：
- **API兼容**：DeepSeek API兼容OpenAI格式，但有特有参数（如`top_p`、`frequency_penalty`的默认值不同）
- **上下文缓存**：利用DeepSeek的上下文缓存功能，减少重复token消耗（长期记忆部分可缓存）
- **中文写作优化**：针对DeepSeek的中文能力，优化各Agent的系统提示词
- **长上下文**：DeepSeek支持128K上下文，可加载更多前文和设定
- **降级策略**：DeepSeek不可用时，自动切换到备选模型

### 前端 (React)
- **框架**：Vite + React（SPA简单配置，本地优先无需SSR）
- **UI库**：Shadcn/ui（轻量、可定制、无重型依赖）
- **富文本编辑器**：TipTap（可扩展、支持协作）
- **状态管理**：Zustand

## 验证计划

1. **后端API**：启动Go服务器，用curl/httpie验证所有REST端点
2. **认证流程**：注册 → 登录 → 用JWT访问受保护端点
3. **前端**：启动开发服务器，验证所有页面渲染并连接后端
4. **构思对话**：与构思Agent多轮对话，验证大纲生成
5. **写作工作流**：生成章节 → 验证内容质量和流式传输
6. **讨论工作流**：开始审稿 → 验证3个Agent响应 → 验证建议汇总
7. **记忆系统**：验证上下文组装包含相关设定/人物
8. **错误处理**：模拟API故障，验证重试和用户反馈
